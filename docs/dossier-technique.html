MIBsoft — Plateforme SSIAP

Dossier technique — architecture, modèle de données et flux

1. Vue d'ensemble

Plateforme web de formation et d'entraînement à la sécurité incendie (SSIAP 1/2/3). Elle sert plusieurs profils — centres de formation, formateurs indépendants, apprenants individuels, formateurs et stagiaires — avec vente en ligne (abonnements et paiement unique), gestion des sessions, et plusieurs moteurs d'entraînement (auto-entraînement QCM/multi-jeux, quiz en salle, Challenge Cup par équipes).

Stack technique

Front : HTML/CSS/JS (vanilla) + Tailwind + Chart.js Hébergement : Vercel Backend : Supabase (PostgreSQL + Auth + RLS) Edge Functions : Deno Paiement : Stripe Emails : Mailgun Realtime jeu : couche compatible Firebase persistée dans Supabase

Architecture multi-tenant : la table centers représente le locataire (centre, indépendant ou apprenant selon license_type). Toutes les données sont cloisonnées par center_id via les politiques de sécurité PostgreSQL (RLS).

7. Sécurité & cloisonnement

  • Authentification : Supabase Auth (email/mot de passe pour centre / indépendant / apprenant ; PIN 6 chiffres hashé SHA-256 pour formateurs et stagiaires).
  • RLS (Row Level Security) activé sur toutes les tables sensibles. Règle centrale : un utilisateur ne voit que les données de son centre (center_id = jwt_center_id()), l'apprenant lit sa fiche via auth_user_id = auth.uid(). Un super-admin (jwt_is_super_admin()) accède à l'ensemble.
  • Paiements : signatures Stripe vérifiées côté edge function ; idempotence des événements (stripe_events).
  • RGPD : consentement enregistré à chaque connexion ; hébergement Europe.

9. Variables d'environnement & secrets

Aucun secret n'est stocké dans le code. Deux niveaux de configuration :

9.1 — Front (valeurs publiques)

La configuration Supabase du front est définie directement dans supabase.js (versionné, lignes ~10–11) : l'URL du projet et la clé publique anon (sans danger côté client — l'accès reste filtré par les politiques RLS). Pour pointer vers un autre projet Supabase, il suffit d'éditer ces deux constantes.

Constante (dans supabase.js)Description
SUPABASE_URLURL du projet Supabase (ex. https://<projet>.supabase.co).
SUPABASE_ANON_KEYClé publique anon du projet (lecture filtrée par RLS).

Note : un fichier config.js est référencé par les pages mais reste optionnel (les valeurs ci-dessus suffisent au fonctionnement).

10. Guide de déploiement

10.1 — Base & backend (Supabase)

  1. Créer le projet Supabase (région UE).
  2. Appliquer les migrations SQL : dossier supabase/migrations/ (via supabase db push ou le SQL Editor).
  3. Déployer les Edge Functions (supabase/functions/) : stripe-create-checkout, stripe-webhook, health-monitor, etc.
    ⚠️ stripe-webhook et stripe-create-checkout doivent être en verify_jwt = false (endpoints publics / webhook).
  4. Renseigner les Secrets (cf. §9.2).
  5. Auth → URL Configuration : ajouter les Redirect URLs autorisées (domaine de prod + *.vercel.app) pour les liens « définir/réinitialiser le mot de passe ».

10.2 — Paiement (Stripe)

  1. Créer les produits & prix (mensuel/annuel/paiement unique selon le plan).
  2. Copier chaque price_id dans la table stripe_prices (plan, cycle, amount_cents, active=true).
  3. Créer un endpoint webhook pointant vers la function stripe-webhook, événements : checkout.session.completed, customer.subscription.updated|deleted, invoice.payment_failed.
  4. Copier le secret de signature du webhook dans STRIPE_WEBHOOK_SECRET.

10.3 — Front (Vercel)

  1. Connecter le dépôt GitHub à Vercel (déploiement auto sur main).
  2. Vérifier la config Supabase du front dans supabase.js (cf. §9.1) — l'éditer si l'on change de projet.
  3. Brancher le domaine (ex. mibsoft.fr) et vérifier qu'il fait partie des origines autorisées (CORS) des edge functions et des Redirect URLs Supabase.

10.4 — Vérifications post-déploiement

  • Achat de test (carte 4242 4242 4242 4242 en mode test) → compte créé + email reçu + connexion OK.
  • Webhook Stripe : événements en 200 (table stripe_events à l'état done).
  • Un entraînement (auto-entraînement / quiz salle / challenge) → résultats bien enregistrés.
